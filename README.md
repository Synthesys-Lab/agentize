# Agentize - AI Development Workflow SDK

An AI-powered development workflow framework that automates issue resolution, code review, and project management through Claude Code agents.

## Overview

Agentize is a project-neutral SDK that provides:
- **Automated Issue Resolution**: End-to-end workflow from GitHub issue to PR
- **Interactive Design Reviews**: Three-agent brainstorming for idea validation
- **Code Quality Enforcement**: Automated code review with scoring
- **Project Management Integration**: Automatic GitHub Project board updates
- **Milestone-Based Work Continuation**: Structured handoff for large tasks

## Quick Start

### Installation Modes

Agentize supports two installation modes:

#### 1. Initialize New Project (`init` mode)

Creates a complete project structure with AI workflow configurations:

```bash
cd agentize/
make agentize \
  AGENTIZE_MASTER_PROJ=/path/to/new-project \
  AGENTIZE_PROJ_NAME="My New Project" \
  AGENTIZE_MODE=init
```

**Creates:**
- `.claude/` - AI workflow configurations (agents, commands, rules, skills, hooks)
- `docs/CLAUDE.md` - Project documentation template
- `Makefile` - Build/test targets
- `README.md` - Project README stub
- `.gitignore` - Common ignore patterns
- `setup.sh` - Environment setup script

#### 2. Port to Existing Project (`port` mode)

Adds AI workflow configurations to an existing project without modifying project files:

```bash
cd agentize/
make agentize \
  AGENTIZE_MASTER_PROJ=/path/to/existing-project \
  AGENTIZE_PROJ_NAME="Existing Project" \
  AGENTIZE_MODE=port
```

**Creates:**
- `.claude/` - AI workflow configurations only

### Configuration Variables

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `AGENTIZE_MASTER_PROJ` | No | `../..` | Path to target project |
| `AGENTIZE_PROJ_NAME` | No | `MyProject` | Project name |
| `AGENTIZE_MODE` | No | `init` | Installation mode: `init` or `port` |
| `AGENTIZE_LANG` | No | (empty → `cpp`) | Languages: `python`, `c`, `cpp`, `rust` (comma-separated) |
| `AGENTIZE_IMPL_DIR` | No | `src` | Implementation directory: `src`, `lib`, etc. |

### Language-Specific Initialization

Agentize automatically generates language-specific project templates with runnable "hello world" examples:

#### Python Project

```bash
make agentize \
  AGENTIZE_MASTER_PROJ=~/my-python-lib \
  AGENTIZE_PROJ_NAME="My Python Library" \
  AGENTIZE_LANG=python \
  AGENTIZE_IMPL_DIR=lib
```

**Generates:**
- `pyproject.toml` - Python package configuration
- `lib/my_python_library/` - Package with `__init__.py` and `main.py`
- `tests/test_main.py` - Basic test suite
- `Makefile` - Unified build wrapper with `make build`, `make test`

**Run:** `python -m lib.my_python_library.main`

#### C++ Project

```bash
make agentize \
  AGENTIZE_MASTER_PROJ=~/my-cpp-project \
  AGENTIZE_PROJ_NAME="My C++ Project" \
  AGENTIZE_LANG=cpp
```

**Generates:**
- `CMakeLists.txt` - CMake build configuration
- `include/my_cpp_project/` - Namespaced public headers
- `src/` - Implementation and entry point
- `tests/` - Test suite with CMake integration
- `Makefile` - Wrapper calling CMake commands

**Build:** `cmake -B build && cmake --build build && ./build/main`

#### Rust CLI Tool

```bash
make agentize \
  AGENTIZE_MASTER_PROJ=~/rust-cli \
  AGENTIZE_PROJ_NAME="Rust CLI" \
  AGENTIZE_LANG=rust
```

**Generates:**
- Uses `cargo init` to create standard Rust project
- `Cargo.toml`, `src/main.rs` - Generated by cargo
- `Makefile` - Wrapper for `cargo build`, `cargo test`

**Run:** `cargo run`

#### Multi-Language Project (Python + C++)

```bash
make agentize \
  AGENTIZE_MASTER_PROJ=~/hybrid-project \
  AGENTIZE_PROJ_NAME="ML Accelerator" \
  AGENTIZE_LANG=python,cpp \
  AGENTIZE_IMPL_DIR=src
```

**Generates:**
- Both Python and C++ project structures
- Unified Makefile that builds both
- Language-specific `.gitignore` patterns
- Integrated test commands

**Use case:** Python bindings with C++ performance-critical modules

## Key Commands

Once installed, use these slash commands in your Claude Code session:

```bash
# Resolve a GitHub issue end-to-end
/issue2impl 123

# Transform an idea into actionable issues
/feat2issue "Add user authentication feature"

# Generate milestone for incomplete work
/gen-milestone

# Create commit following project standards
/git-commit

# Update related issues in the dependency graph
/update-related-issues 123
```

## Workflow Overview

### `/issue2impl` - 9-Phase Workflow

1. **Input Validation** - Verify issue number, branch, dependencies
2. **Issue Analysis** - Analyze requirements and codebase (with triage)
3. **Documentation** - Interactive documentation review
4. **Planning** - Plan implementation with size estimates
5. **Implementation** - Code changes with size monitoring
6. **Code Review** - Skeptical review with scoring (target: ≥81/100)
7. **Commit & PR** - Verify build, commit, create/update PR
8. **Remote Review** - Handle PR feedback cycles
9. **Finalize** - Update related issues, generate summary

**Triage System:**
- **Fast**: Single file, doc-only, <50 lines → Skip docs/planning
- **Standard**: Default full workflow
- **Extended**: Multi-component, >1500 lines → Add architecture review

### `/feat2issue` - Design to Implementation

1. **Input Validation** - Parse idea (text or file)
2. **Three-Agent Brainstorming**:
   - `idea-creative-proposer` → Generate alternatives
   - `idea-critical-checker` → Fact-check and critique
   - `idea-comprehensive-analyzer` → Synthesize recommendation
3. **Issue Research** - Find related existing issues
4. **Documentation Research** - Analyze docs state
5. **Planning** - Plan docs PR + implementation issues
6. **Implementation** - Create docs PR + implementation issues
7. **Cleanup** - Update related issues

## Customization

The SDK is highly customizable. After installation:

1. **Customize Component Tags** in `.claude/git-tags.md`:
   ```markdown
   # Git Component Tags

   ## Component Tags

   - [CORE] - Core business logic
   - [API] - REST API endpoints
   - [UI] - User interface components
   ```

2. **Add Project-Specific Rules** in `.claude/rules/`:
   - `custom-project-rules.md` - Coding conventions, testing requirements
   - `custom-workflows.md` - Development workflows, deployment processes

3. **Configure Build Commands** in your `Makefile`:

   For Python projects:
   ```makefile
   build:
       pip install -e .

   test:
       pytest tests/
   ```

   For C/C++ projects:
   ```makefile
   build:
       cmake -B build && cmake --build build

   test:
       cd build && ctest --output-on-failure
   ```

   For Rust projects:
   ```makefile
   build:
       cargo build --release

   test:
       cargo test
   ```

See `claude/CUSTOMIZATION_GUIDE.md` for detailed instructions.

## Features

### Automated Workflows
- **Input Validation**: Verify issue state, dependencies, branch naming
- **Documentation Review**: Interactive brainstorming with `doc-architect` agent
- **Code Review**: Skeptical review with scoring system (target: ≥81/100)
- **Size Monitoring**: Automatic milestone creation when PR exceeds thresholds
- **Project Board Integration**: Every issue automatically added to GitHub Project

### Agent Architecture
- **13 Specialized Agents**: From input validation to project management
- **Multi-Agent Brainstorming**: Three-agent chain for design validation
- **Model Selection**: Haiku for fast validation, Opus for critical thinking
- **Subagent Coordination**: Main thread orchestrates agent spawning

### Customization Points
- **Component Tags**: Define L1/L2 tags for your project structure
- **Custom Rules**: Extend with project-specific conventions
- **Build Integration**: Adapt to any build system (npm, cargo, make, etc.)
- **Triage Thresholds**: Adjust fast/standard/extended criteria

## Requirements

- **Claude Code CLI**: The Anthropic Claude Code command-line tool
- **Git**: Version control
- **GitHub CLI (`gh`)**: For issue/PR management
- **Make**: Build automation

## SDK Development

### File Organization

```
agentize/
├── Makefile                    # Installation interface
├── scripts/install.sh          # Installation script
├── claude/                     # SDK components
│   ├── README.md               # Component documentation
│   ├── CUSTOMIZATION_GUIDE.md  # Customization guide
│   ├── agents/                 # 13 specialized agents
│   ├── commands/               # 8 slash commands
│   ├── rules/                  # 12 behavioral rules
│   ├── skills/                 # 4 reference skills
│   ├── hooks/                  # 3 lifecycle hooks
│   └── templates/              # 8 project templates
└── templates/                  # Language-specific project templates
    ├── python/                 # Python project template
    ├── cxx/                    # C++ project template
    └── c/                      # C project template
```

### Template Development

Templates use placeholder substitution:
- `${PROJECT_NAME}` → `AGENTIZE_PROJ_NAME`
- `__NAME__` → Project name in snake_case (e.g., `my_project`)
- `__NAME_UPPER__` → Project name in UPPER_SNAKE_CASE (e.g., `MY_PROJECT`)
- `__PROJECT_NAME__` → Project name alphanumeric for CMake (e.g., `MyProject`)

Language-specific templates are in `templates/python/`, `templates/cxx/`, `templates/c/`.

See `scripts/install.sh` for substitution logic.

### Testing Strategy

**IMPORTANT**: Always use `make agentize` to run the installation script. DO NOT run `scripts/install.sh` directly with bash.

Test by installing into sample projects:
```bash
# Test init mode
make agentize AGENTIZE_MASTER_PROJ=/tmp/test-init AGENTIZE_MODE=init

# Test port mode
make agentize AGENTIZE_MASTER_PROJ=/tmp/test-port AGENTIZE_MODE=port

# Test multi-language projects
make agentize AGENTIZE_MASTER_PROJ=/tmp/test-multi AGENTIZE_LANG=python,cpp AGENTIZE_MODE=init

# Verify installation
ls -la /tmp/test-init/.claude
ls -la /tmp/test-port/.claude

# Verify generated Makefile for multi-language project
cat /tmp/test-multi/Makefile
```
